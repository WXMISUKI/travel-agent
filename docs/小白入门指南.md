## 一、什么是旅行规划智能体？

### 1.1 简单的比喻

想象一下，你有一个**超级聪明的秘书**。

当你对她说："帮我查一下后天去杭州要穿什么、带什么？"

这个秘书会：
1. 🧠 先理解你的问题 → 你是想去杭州旅游
2. 📅 算一下后天是什么日期 → 3月16日
3. 🌤️ 查一下杭州后天的天气 → 多云，15-22度
4. 🎯 想想这个季节杭州有什么好玩 → 西湖、灵隐寺...
5. 📝 然后整理成你看得懂的回复 → "后天杭州天气不错，建议..."

**我们做的就是这个"智能秘书"！**

只不过它不是人，而是一个**电脑程序**，但它能听懂人话、会自己查资料、还能帮你整理信息。

### 1.2 它能做什么？

这个智能体可以帮你：

| 你说的话 | 它帮你做的事 |
|----------|-------------|
| "明天天气怎么样" | 查天气预报 |
| "北京到上海的高铁票" | 查火车票余票 |
| "杭州有什么好玩" | 推荐景点 |
| "帮我规划一个行程" | 制定旅行计划 |

## 二、整体架构 - 像搭积木一样

### 2.1 想象一个工厂流水线

把这个智能体想象成一个**工厂流水线**：

```
用户输入 → [处理1] → [处理2] → [处理3] → 最终回复
          (理解问题) (查资料) (整理答案)
```

**每个环节各司其职：**

1. **理解问题环节**：听明白用户想要什么（查天气？买票？）
2. **查资料环节**：根据需要去调用各种工具（天气API、火车票API等）
3. **整理答案环节**：把查到的信息变成人话，回复给用户

### 2.2 实际的技术架构

虽然上面说的是流水线，但实际代码里我们把它分成了**几层**：

```
┌─────────────────────────────────────┐
│         用户界面层                  │
│   (你看到的网页、对话框)            │
└─────────────────────────────────────┘
              ↓ 发送问题
┌─────────────────────────────────────┐
│         API服务层                    │
│   (接收请求、分发给Agent)            │
└─────────────────────────────────────┘
              ↓ 调用
┌─────────────────────────────────────┐
│       Agent智能核心层  ⭐            │
│   (理解问题、规划任务、调用工具)    │
└─────────────────────────────────────┘
              ↓ 调用
┌─────────────────────────────────────┐
│        工具和数据源层                │
│   (天气API、火车票API、搜索API)     │
└─────────────────────────────────────┘
```

### 2.3 每一层具体做什么？

**第1层：用户界面（Frontend）**

你看到的那个网页对话框，就是用户界面。

它的作用很简单：
- 显示你的问题
- 显示智能体的回复
- 让你和智能体可以"聊天"

**第2层：API服务（FastAPI）**

这部分是**传话员**。

你的问题先发给它，它再转发给后面的"大脑"（Agent），等大脑处理完，它再把结果传回给你。

**第3层：Agent核心 ⭐这是最重要的**

这是整个系统的**大脑**，分几个部分：

| 模块 | 作用 | 类比 |
|------|------|------|
| 意图识别 | 判断你想做什么 | 听明白"要查天气" |
| 实体提取 | 提取关键信息 | 提取"杭州"、"后天" |
| 执行计划 | 规划怎么做 | "先查天气，再查景点" |
| 工具执行 | 实际调用API | 真的去查天气数据 |
| 响应生成 | 组织语言回复 | 把结果写成话 |

**第4层：工具和数据源**

这些是**手脚**，帮大脑做事：

| 工具 | 能做什么 |
|------|----------|
| 天气查询 | 查某城市某日期的天气 |
| 火车票查询 | 查某线路的余票 |
| 景点搜索 | 搜某城市有什么好玩的 |
| 通用搜索 | 百度搜索（兜底用） |

## 三、核心流程 - 它是如何思考的？

### 3.1 一个完整的例子

假设你对它说：

> **"我想去杭州玩三天，帮我查一下天气和景点"**

它内部是这样"思考"的：

```
Step 1: 理解用户要什么
┌─────────────────────────────────────┐
│ 用户说：我想去杭州玩三天...          │
│                                     │
│ 意图识别结果：旅行规划 (trip_plan)   │
│ 目的地：杭州                        │
│ 关键词：景点                        │
└─────────────────────────────────────┘
              ↓
Step 2: 规划要做哪些事
┌─────────────────────────────────────┐
│ 生成执行计划：                      │
│ [1] 先解析"三天"是哪天              │
│ [2] 查杭州那几天的天气              │
│ [3] 搜索杭州的景点                  │
│ [4] 搜索杭州旅游攻略                │
└─────────────────────────────────────┘
              ↓
Step 3: 执行计划（一个一个来）
┌─────────────────────────────────────┐
│ [1] 解析日期：3月14日→3月16日       │
│ [2] 查天气：调用天气API             │
│ [3] 搜景点：调用搜索API             │
│ [4] 搜攻略：调用百度搜索            │
└─────────────────────────────────────┘
              ↓
Step 4: 整理回复
┌─────────────────────────────────────┐
│ 根据所有查询结果，生成最终回复：    │
│                                     │
│ 您好！根据您的需求，我帮您查询到：  │
│                                     │
│ 📅 日期：3月14日-3月16日           │
│ 🌤️ 天气：杭州期间天气晴好           │
│                                     │
│ 🎯 推荐景点：                       │
│ • 西湖 - 必去！                     │
│ • 灵隐寺 - 求签祈福                 │
│ • 宋城 - 看演出                     │
│ ...                                 │
└─────────────────────────────────────┘
```

### 3.2 意图识别 - 它怎么知道我要什么？

**方法：关键词匹配 + AI辅助**

就像你教一个小孩认识"什么是苹果"：

```
苹果的特征：圆的、红色的、甜的...
看见符合这些特征的，就知道是苹果
```

我们定义了很多**关键词模式**：

| 意图 | 关键词 |
|------|--------|
| 天气 | 天气、温度、冷、热、下雨 |
| 火车票 | 火车票、高铁、动车、坐火车 |
| 景点 | 景点、好玩、旅游、网红 |
| 旅行规划 | 规划、行程、攻略、去玩 |

当用户说"查天气"，包含"天气"这个词，就知道用户想查天气。

### 3.3 实体提取 - 它怎么提取关键信息？

你说"明天北京到上海的高铁票"

它需要提取：
- **时间**：明天
- **出发地**：北京
- **目的地**：上海
- **类型**：高铁

这就像做"完形填空"，从一句话里把关键信息抠出来。

### 3.4 降级策略 - 如果查不到怎么办？

有时候API会罢工（比如天气服务器崩溃了），这时候怎么办？

**我们的策略：降级兜底**

```
正常流程：天气API → 查不到 → 报错

降级流程：
  天气API → 查不到 → 百度搜索天气 → 还是有结果的！
```

就像你问路：
- 首选：看地图（精准）
- 备选：问路人（可能不太准但能用）
- 最后：自己猜（兜底）

## 四、关键技术概念 - 用比喻来理解

### 4.1 什么是LLM（大型语言模型）？

**比喻：**

LLM就像一个**读了很多书的学霸**。

- 它读过海量的文字（训练数据）
- 你问它问题，它能理解
- 它能"编"出合理的回答

我们用的" MiniMax M2.5 "就是这样一个学霸，不过它是专门优化过的，更擅长中文。

### 4.2 什么是LangChain/LangGraph？

**比喻：**

如果LLM是**大脑**，那LangChain/LangGraph就是**神经系统**。

- 大脑（LLM）会思考
- 神经系统（LangChain）负责把思考变成行动
  - 让大脑去调用工具
  - 让大脑记住之前说了什么
  - 让大脑按流程一步步来

### 4.3 什么是API？

**比喻：**

API就像**餐厅的服务员**。

```
你（程序） → 点菜（发送请求） → 服务员（API）
                                         ↓
                              厨房（外部服务/数据库）
                                         ↓
                              上菜（返回结果） → 你（程序）
```

你不需要知道厨房怎么做菜，只需要通过服务员点单，就能得到结果。

### 4.4 什么是JSON？

**比喻：**

JSON就像**表格**，一种固定格式来组织数据。

```json
{
    "城市": "杭州",
    "天气": "晴",
    "温度": 22
}
```

就像这样：
| 城市 | 天气 | 温度 |
|------|------|------|
| 杭州 | 晴 | 22 |

## 五、为什么这样设计？

### 5.1 分层设计的好处

**想象一下：**

如果把所有代码混在一起，就像把所有食材都扔进一个锅，做出来的东西肯定不好吃。

但如果**分层处理**：
- 洗菜归洗菜
- 切菜归切菜
- 炒菜归炒菜

每个环节专注做一件事，最后组装起来，既清晰又容易维护。

### 5.2 为什么要这么多数据源？

**因为没有"万能的"数据源！**

就像：
- 查天气 → 用专业天气服务（Open-Meteo）
- 查火车票 → 用12306官方服务
- 查景点 → 用百度搜索

每个服务各有所长，我们把它们组合起来，给用户最好的体验。

### 5.3 为什么要做降级？

**因为网络服务不稳定！**

再好的服务也可能挂，就像：
- 天气API可能临时不可用
- 火车票服务器可能崩溃

如果我们没有降级方案，这时候用户就会收到"系统错误"的反馈，体验很差。

有了降级，至少能给用户一个**可用**的答案。

## 六、怎么运行这个项目？

### 6.1 前置要求

你需要准备：

1. **Python环境**（电脑上安装Python 3.11或更高版本）
2. **MiniMax API Key**（去阿里云DashScope申请）
3. **一点耐心**（跟着步骤走）

### 6.2 启动步骤

**Step 1：安装依赖**

打开命令行（CMD或终端），输入：

```bash
pip install -r requirements.txt
```

**Step 2：配置API Key**

编辑 `.env` 文件，填入你的API Key：

```
ORCH_API_KEY=你申请的Key
```

**Step 3：启动服务**

```bash
uvicorn src.main:app --reload --port 8000
```

**Step 4：打开网页**

浏览器访问：`http://localhost:8000`

## 七、常见问题

### Q1：这个智能体是真人吗？

**不是**，是程序。但它背后有一个很强大的AI模型（MiniMax），能理解你的问题并回答。

### Q2：它是怎么"学会"查天气、查火车票的？

**它不是"学会"的**，是我们把"怎么查"的代码写好，它只是负责调用这些代码。

就像你教一个小孩：
- "查天气"就是"调用天气API"
- "查火车票"就是"调用火车票API"

### Q3：为什么有时候回答得不好？

可能有几个原因：
1. 你的问题太复杂，AI理解有偏差
2. 某个API暂时不可用
3. 需要更多训练数据来优化

### Q4：这个和Siri、小爱同学有什么区别？

原理上差不多，都是"理解人话、执行操作"。

区别在于：
- Siri、小爱是**通用助手**，什么都能干
- 我们的智能体是**垂直领域**，专门做旅行规划，在旅行这块更专业
