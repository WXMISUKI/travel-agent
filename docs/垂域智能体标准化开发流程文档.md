# 垂域智能体标准化开发流程文档

## 文档说明

本文档定义了**垂域智能体（垂直领域 AI Agent）** 从需求到落地的通用、成熟开发流程，适用于如番剧查询、电商客服、财务分析、医疗咨询等各类垂直场景，覆盖「需求→设计→开发→测试→部署→运维」全生命周期，可作为你开发不同智能体的标准化操作指南。

## 二、各阶段详细操作指南

### 阶段 1：需求分析与拆解（基础核心，避免返工）

#### 1.1 核心目标

明确智能体的**边界、价值、用户场景**，把模糊需求转化为可落地的量化指标。

#### 1.2 关键步骤

表格







|      步骤      |                           具体操作                           |             输出物             |
| :------------: | :----------------------------------------------------------: | :----------------------------: |
| 1.2.1 需求收集 | 1. 访谈目标用户 / 产品方，明确「用户是谁、解决什么问题、使用场景」2. 梳理核心需求（如「番剧智能体」：查询最新番剧、筛选番剧类型、推送更新提醒）3. 区分「核心需求」（必须实现）和「非核心需求」（后续迭代） |    《需求清单》（含优先级）    |
| 1.2.2 需求量化 | 1. 定义核心指标（KPI）：   - 功能指标：响应时间 <3s、查询准确率> 90%   - 体验指标：用户满意度 > 85%、错误率 < 5%2. 明确输入 / 输出格式（如输入：自然语言问题；输出：结构化番剧信息 + 自然语言回答） |   《需求规格说明书（SRS）》    |
| 1.2.3 场景拆解 | 1. 拆分核心场景（如番剧智能体拆分为：   - 场景 1：解析用户查询意图   - 场景 2：调用番剧数据源 API   - 场景 3：格式化回答返回用户）2. 梳理异常场景（如数据源失效、用户问题模糊） | 《场景拆解文档》（含异常场景） |

#### 1.3 工具 / 方法

- 需求收集：飞书 / Notion（记录）、用户访谈纪要模板

- 需求拆解：用户故事（User Story）、用例图（Draw.io/Figma）

- 示例（番剧智能体用户故事）：

  

  ```
  作为用户，我希望输入「2026年3月最新日漫」，智能体能返回包含名称、更新时间、平台的番剧列表，且响应时间不超过3秒。
  ```

### 阶段 2：技术选型与方案设计

#### 2.1 核心目标

根据需求选择适配的技术栈，设计可落地的技术方案，避免重复造轮子。

#### 2.2 关键步骤

表格







|          模块           |                           选型维度                           |                       成熟方案 / 建议                        |
| :---------------------: | :----------------------------------------------------------: | :----------------------------------------------------------: |
|    2.2.1 大模型选型     | 1. 场景适配：垂域数据量 / 推理复杂度2. 成本：API 调用费 / 私有化部署成本3. 兼容性：是否支持标准接口（OpenAI 兼容）4. 能力：意图理解 / 结构化输出能力 | - 通用场景：MiniMax/M2.5、GPT-3.5/4o、文心一言- 私有化：通义千问私有化版、LLaMA-2（微调）- 选型原则：先轻量 API 版验证，再考虑私有化 |
|   2.2.2 开发框架选型    | 1. 智能体能力：是否需要工具调用 / 记忆 / 多轮对话2. 开发效率：是否有成熟组件（如技能封装）3. 语言适配：Python/JS（垂域技术栈） | - 基础版：纯 Python（requests+openai 库）- 进阶版：LangChain（工具调用 / 工作流）、LlamaIndex（数据接入）、AutoGPT（自主决策）- 前端交互：FastAPI（后端）+ Vue/React（前端） |
| 2.2.3 工具 / 数据源选型 | 1. 垂域数据源：是否有开放 API / 结构化数据2. 工具能力：是否需要搜索 / 计算 / 存储3. 合规性：数据隐私（如番剧数据是否可商用） | - 数据源：开放 API（Bangumi、B 站）、爬虫（合规前提下）、自有数据库- 工具：SerpAPI（通用搜索）、SQLAlchemy（数据库操作）、Redis（缓存） |
|     2.2.4 架构设计      | 1. 分层设计：接入层 / 推理层 / 数据层 / 工具层2. 流程设计：用户输入→意图解析→工具调用→结果格式化→返回3. 容错设计：降级策略（如模型不可用时返回默认回答） |       《架构设计文档》（含流程图）、《技术方案选型表》       |

#### 2.3 工具 / 输出物

- 选型工具：对比表格（Notion/Excel）、模型能力测试脚本
- 设计工具：Draw.io（架构图 / 流程图）、Mermaid（文本流程图）
- 核心输出：《技术方案文档》（含选型理由 + 架构图 + 核心流程）

# 模型客户端封装示例
from openai import OpenAI
from dotenv import load_dotenv
import os
load_dotenv()

class LLMClient:
    def __init__(self):
        self.client = OpenAI(
            api_key=os.getenv("ORCH_API_KEY"),
            base_url=os.getenv("ORCH_API_BASE")
        )
    def chat(self, prompt, model="MiniMax/MiniMax-M2.5"):
        try:
            resp = self.client.chat.completions.create(
                model=model,
                messages=[{"role":"user","content":prompt}]
            )
            return resp.choices[0].message.content
        except Exception as e:
            self.logger.error(f"模型调用失败：{e}")
            return "暂无法回答，请稍后重试"

#### 3.3 开发规范
1. 代码分层：`config/`（配置）、`core/`（核心逻辑）、`skills/`（技能）、`utils/`（工具）、`tests/`（测试）
2. 版本控制：Git提交规范（如feat: 新增番剧筛选技能、fix: 修复数据源调用异常）
3. 敏感信息：绝对不硬编码API Key，使用.env+gitignore

### 阶段4：测试与调优
#### 4.1 核心目标
验证智能体功能、性能、体验是否达标，通过调优提升效果。
#### 4.2 测试维度与方法
| 测试类型 | 测试内容 | 方法/工具 | 验收标准 |
|----------|----------|-----------|----------|
| 4.2.1 功能测试 | 1. 核心功能是否实现<br>2. 异常场景是否处理<br>3. 输入输出是否符合预期 | 手动测试（测试用例）、自动化测试（pytest） | 核心功能100%覆盖，异常场景无崩溃 |
| 4.2.2 性能测试 | 1. 响应时间<br>2. 并发处理能力<br>3. 模型调用成功率 | Locust（压测）、手动计时 | 响应时间<3s，并发100用户时成功率>95% |
| 4.2.3 效果调优 | 1. 意图理解准确率<br>2. 回答准确率/流畅度<br>3. 用户体验 | 人工标注测试集、A/B测试（不同Prompt/模型） | 准确率>90%，用户反馈满意度>85% |
| 4.2.4 合规测试 | 1. 数据隐私（如用户数据是否泄露）<br>2. 内容合规（如无违规信息） | 人工审核、合规扫描工具 | 无隐私泄露，内容符合规范 |

#### 4.3 调优方向
1. Prompt调优：优化指令（如明确输出格式、降低温度参数提升稳定性）
2. 模型调优：小样本微调（垂域数据充足时）、Prompt Engineering（零样本）
3. 工具调优：缓存高频查询结果（Redis）、优化数据源调用方式（批量查询）
4. 体验调优：简化回答、补充上下文解释

### 阶段5：部署与发布
#### 5.1 部署方案选型（按复杂度排序）
| 部署方式 | 适用场景 | 工具/步骤 |
|----------|----------|-----------|
| 轻量部署 | 个人/小团队、测试阶段 | 1. 本地运行→暴露API（FastAPI）<br>2. 内网穿透（ngrok）供测试 |
| 容器部署 | 正式环境、多环境隔离 | 1. 编写Dockerfile<br>2. 构建镜像：docker build -t anime-agent .<br>3. 运行容器：docker run -p 8000:8000 anime-agent |
| 云服务部署 | 高可用、规模化 | 1. 云服务器（ECS/阿里云）/Serverless（AWS Lambda/阿里云函数计算）<br>2. 配置域名/HTTPS<br>3. 负载均衡（高并发时） |
| 私有化部署 | 企业级、数据敏感 | 1. 模型私有化部署（如MiniMax私有化版）<br>2. 内网部署服务，禁止公网访问 |

#### 5.2 发布流程
1. 灰度发布：先给10%用户使用，监控问题
2. 全量发布：无重大问题后全量上线
3. 发布文档：编写《用户使用手册》《接口文档》

### 阶段6：运维与迭代
#### 6.1 运维监控
1. 监控指标：调用量、成功率、响应时间、错误率
2. 监控工具：Prometheus+Grafana（可视化）、日志平台（ELK）
3. 告警机制：错误率>5%时触发邮件/钉钉告警

#### 6.2 迭代优化
1. 数据收集：用户反馈、使用日志、错误案例
2. 迭代计划：按优先级迭代非核心需求（如番剧智能体新增「番剧评分查询」）
3. 版本更新：语义化版本（如v1.0.0→v1.1.0），保留回滚能力

## 四、核心注意事项
1. 合规性：垂域数据需确保商用授权（如番剧数据、医疗数据），用户隐私需脱敏
2. 可扩展性：技能（Skill）设计遵循「单一职责」，便于后续新增/修改
3. 成本控制：优先使用API版模型验证，私有化部署需评估成本收益
4. 降级策略：模型/数据源不可用时，需有兜底方案（如返回默认回答）

### 总结
1. 垂域智能体开发核心遵循「需求拆解→技术选型→模块化开发→全维度测试→工程化部署→持续迭代」六阶段，每个阶段需输出明确文档/代码，避免模糊开发；
2. 技术选型优先「成熟轻量方案」（如Python+LangChain+MiniMax API），验证效果后再升级为复杂架构；
3. 核心规范：模块化拆分技能、环境隔离、敏感信息保护、全流程测试，是保证智能体稳定落地的关键。

该流程可直接复用至任意垂域智能体开发（如电商客服、财务分析），仅需替换「垂域数据源/技能逻辑」，框架和流程完全通用。
